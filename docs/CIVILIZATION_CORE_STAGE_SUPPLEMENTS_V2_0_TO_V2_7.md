# 文明之核：v2.0-v2.7 阶段治理补章

This document is not a new capability release. It is a stage-by-stage governance supplement that records what each previous phase still needed to clarify under the Civilization Core - Subspace Memory System goal.

本文不是新的能力发布，而是一份逐阶段治理补章，用于记录在 “Civilization Core - Subspace Memory System” 目标下，各既有阶段仍需澄清的治理边界。

状态：v2.8-pre-2 文档补章
范围：阶段 0、阶段 1、v2.0、v2.1-v2.2、v2.3、v2.4、v2.5、v2.6、v2.7
性质：治理说明，不新增工具、不新增执行能力、不改变 release tag

本补章不宣称系统已经完成。它只把此前被压缩在总览中的阶段性治理缺口逐一拆开，明确每个阶段已经完成什么、仍然缺什么、以及进入 v2.8.0 前不得越过哪些边界。

## 阶段 0：文明之核总宪法缺口

### 原始阶段定位

阶段 0 是文明之核的总宪法定位阶段。它应当回答本仓库为什么存在、系统治理对象是什么、哪些边界高于自动化便利性。

在既有阶段中，这一层长期以隐含目标存在：Hermes、Codex CLI、OpenClaw、GitHub 和人类操作者各自承担治理链条的一部分，但缺少明确的最高层定义。

### 已完成内容

既有对齐文档已经把项目重新定位为 “Civilization Core - Subspace Memory System”，并说明本仓库不是单一插件、工具或脚本集合，而是长期治理内核的一个实现面。

该文档已经提出记忆、证据、执行隔离、审计责任链、多智能体协作和版本封印等核心方向。

### 发现的缺陷

在对齐文档之前，项目没有显式的顶层宪法文本。阶段性交付物虽能证明若干治理构件存在，但不能单独说明这些构件受同一个文明之核目标约束。

缺少顶层宪法时，后续功能容易被误读为自动化工具扩展，而不是治理内核建设。

### 本补章补齐内容

文明之核应被定义为治理内核，而不是自动化工具包。

该治理内核至少覆盖以下边界：

- 记忆边界：召回、候选记忆、durable memory、图谱写入必须区分。
- 证据边界：证据候选、审计事实、治理结论必须区分。
- 批准边界：批准请求、批准意图、批准记录、批准令牌必须区分。
- 审计边界：观察、风险说明、通过记录不得自行升级为执行授权。
- 执行边界：执行能力必须受当前、明确、独立的授权约束。
- 版本封印边界：PR、commit、tag、release 记录应作为治理事实，不得被临时状态覆盖。

### 不可越界规则

治理先于自动化。

任何便利性、执行效率或工具能力都不得优先于记忆、证据、批准、审计、执行和版本封印的边界。

### 对 v2.8.0 的约束

v2.8.0 不得以扩展自动化作为起点。它必须先证明最小治理闭环能够在不扩大执行权的条件下成立。

## 阶段 1：亚空间记忆入口契约缺口

### 原始阶段定位

阶段 1 是 Hermes Memory Fabric 作为亚空间记忆入口的契约阶段。它应当说明 Hermes 可以读取什么、可以提出什么、不得直接写入什么，以及召回输出应如何被治理链条解释。

### 已完成内容

既有文档已经把 Hermes Memory Fabric 定位为记忆治理大脑，负责证据、候选记忆、提案、评审、审批请求和审计状态的组织与表达。

文档也已经说明 Hermes 不得绕过治理闸门直接把历史上下文变成 durable memory、图谱写入、执行授权或配置变更。

### 发现的缺陷

Hermes Memory Fabric 已被视为可用的记忆入口，但入口契约不够显式。若不补齐该契约，召回结果可能被误读为已被采纳的长期记忆。

同样，候选证据、历史上下文、策略建议和 durable memory 之间的状态差异需要在阶段层面写清楚。

### 本补章补齐内容

Hermes 可以读取受允许范围内的历史上下文、操作记录、治理文档、候选证据和审计摘要。

Hermes 可以提出候选记忆、候选证据、治理建议、风险说明、写入提案或策略提案。

Hermes 不得直接写 durable memory，不得直接写 Memory Graph，不得直接修改配置，不得直接触发 OpenClaw 执行，不得把召回结果标记为已治理事实。

Hermes 的输出在进入评审、批准或写入流程之前，只能被视为证据候选、分析材料或提案材料。

### 不可越界规则

召回不是记忆写入。

### 对 v2.8.0 的约束

v2.8.0 的自然语言任务可以使用 Hermes 召回作为上下文来源，但不得把召回内容直接当作 durable memory、批准记录或执行命令。

## v2.0：权能分离原则补章

### 原始阶段定位

v2.0 建立令牌权限边界契约 dry-run，用于区分令牌来源、执行材料和权限声明。

该阶段的技术目标是证明系统能够识别和描述权限边界，而不是授予真实执行权。

### 已完成内容

v2.0 已形成 token authority boundary contract dry-run。它证明系统可以围绕 token、authority、capability 和 execution material 建立结构化边界说明。

### 发现的缺陷

技术边界已经存在，但其文明之核含义仍需显式化。若只从工具层理解 token 或 API 能力，容易把“系统能做”误解为“系统被允许做”。

### 本补章补齐内容

能力不是权限。

工具可用、模型可用、令牌来源存在、API 能力存在、终端可以运行命令，都不自动授予执行、写入、合并、打 tag、发布或现实世界操作权限。

权限必须来自独立、明确、当前有效、可审计的授权链条。能力只能说明系统具有潜在操作面，不能说明当前任务已获得使用该操作面的治理许可。

### 不可越界规则

Capability is not authority.

### 对 v2.8.0 的约束

v2.8.0 可以验证 Codex CLI 和终端 / OpenClaw 的受控协作，但不得把可调用工具、可用 token 或可运行命令解释为合并、标签、发布或现实世界执行授权。

## v2.1-v2.2：外部能力摄入协议补章

### 原始阶段定位

v2.1 建立 shared skill fabric，v2.2 建立 local archive import simulation。两个阶段共同涉及外部能力、技能、归档和借用系统如何进入文明之核治理视野。

### 已完成内容

v2.1 已记录共享技能织物的职责边界，强调技能共享必须保留来源、角色和适用范围。

v2.2 已通过本地归档导入模拟说明归档摄入必须先模拟，不能直接污染长期记忆或生产系统。

### 发现的缺陷

Skill Fabric 和 local archive simulation 已经存在，但外部能力摄入规则尚未完整陈述。

外部技能、归档材料和借用系统可能被误认为已经可信、已经安装、已经生产可用，或已经获得执行许可。

### 本补章补齐内容

导入不是信任。

归档不是安装。

模拟通过不是生产启用。

外部技能、归档和借用系统必须通过沙箱化、可审计、可回滚的路径进入。其来源、版本、权限面、风险、适用范围和治理状态必须被记录。未经评审和明确启用前，它们只能作为候选材料、参考材料或模拟对象。

### 不可越界规则

外部能力必须先隔离，再谈信任。

### 对 v2.8.0 的约束

v2.8.0 不得以引入更多外部能力作为闭环验证前提。若验证需要外部技能或归档材料，必须保持 sandboxed、read-only 或 dry-run 状态，并明确其非生产性质。

## v2.3：版本封印制度补章

### 原始阶段定位

v2.3 建立 release integrity audit，用于检查发布完整性、版本状态和回滚边界。

该阶段把 release 从工程结果提升为治理事实的一部分。

### 已完成内容

v2.3 已形成发布完整性审计，证明系统可以围绕 PR、commit、tag 和 release 进行一致性检查。

既有对齐文档也已把 GitHub 定位为版本链、PR、标签、回滚和发布封印的承载面。

### 发现的缺陷

发布完整性审计已经存在，但版本 tag 的文明之核解释仍需加强。若 tag 被视为普通可移动标记，治理历史将失去稳定锚点。

### 本补章补齐内容

PR、commit、tag 和 release audit 均为治理事实。

已发布 tag 不得重写。后续发现的新问题、新风险、新校准和新补章必须进入下一个版本或预发布文档，不得通过改写既有 tag 来覆盖历史。

版本封印记录的是当时已经审计和发布的治理状态；它不表示系统永远正确，也不表示后续无需补章。

### 不可越界规则

版本封印是治理边界。

### 对 v2.8.0 的约束

v2.8.0 必须尊重既有 release tag，不得通过移动、重写或替换旧 tag 来制造连续性。新发现只能进入 v2.8.0 或后续版本链。

## v2.4：候选记忆与提案包边界补章

### 原始阶段定位

v2.4 建立 governed proposal pack dry-run，用于把候选记忆、证据和写入意图整理为可评审材料。

### 已完成内容

v2.4 已证明系统可以生成受治理提案包，并把候选记忆从普通文本中分离出来，形成可标识、可审查、可追溯的材料。

### 发现的缺陷

提案包已经存在，但候选记忆与实际记忆的边界需要更尖锐。候选记忆一旦被写入文档或提案包，容易被误认为已经成为 durable memory。

### 本补章补齐内容

候选记忆不是 durable memory。

提案包是评审材料，不是写入执行。它可以包含来源、理由、证据、范围、风险、建议写入内容和治理状态，但它本身不改变长期记忆状态。

任何 durable memory 写入都必须经过独立的治理写入流程，并留下可复核的批准和写入记录。

### 不可越界规则

Proposal is not write.

### 对 v2.8.0 的约束

v2.8.0 可以使用提案包作为 ChatGPT review 或治理交接材料，但不得把提案包存在本身解释为记忆已经写入。

## v2.5：评审闸门边界补章

### 原始阶段定位

v2.5 建立 governed proposal review gate dry-run，用于对提案包进行结构化评审，判断其是否可进入下一治理步骤。

### 已完成内容

v2.5 已证明系统可以记录 review pass、review reject、risk note 等评审状态，并让评审结果具备可审计结构。

### 发现的缺陷

评审闸门已经存在，但 review outcome 可能被误读为执行许可。结构合格、风险可解释或评审通过并不等同于写入、执行、发布或外部暴露授权。

### 本补章补齐内容

Review pass 只表示提案在结构上可接受，可以进入下一治理步骤。

Review reject 保留证据和拒绝理由；它不会擦除历史记录，也不应删除已形成的审计事实。

Risk note 不等于 block，除非当前政策明确规定该风险类型必须阻断后续步骤。

评审结论必须保持其原始语义：它是治理判断，不是执行命令。

### 不可越界规则

评审不是执行许可。

### 对 v2.8.0 的约束

v2.8.0 的 ChatGPT review 可以形成通过、拒绝或风险说明，但不得直接触发 Codex 合并、GitHub tag、OpenClaw 执行或 Hermes 写入。

## v2.6：批准请求边界补章

### 原始阶段定位

v2.6 建立 governed approval request dry-run，用于表达批准请求信封、审批材料和敏感批准短语边界。

### 已完成内容

v2.6 已证明系统可以构造审批请求 dry-run，并明确审批请求不是批准令牌。

### 发现的缺陷

批准请求、批准意图、批准记录和批准令牌仍需严格分离。若不分离，系统可能把“请求批准”或“记录了批准意图”误解为已经获得可执行授权。

### 本补章补齐内容

批准请求不是批准令牌。

批准意图不是执行许可。

批准记录是治理事实，但仍不等于可执行令牌。执行必须依赖当前有效、作用域明确、可验证、未泄露的批准令牌或等价授权机制。

approval phrase 属于敏感材料，不得泄露到评审工具、stdout、stdout_tail、审计观察窗或不需要知道该短语的文档中。

### 不可越界规则

批准请求不是授权。

### 对 v2.8.0 的约束

v2.8.0 可以验证批准请求如何被审查和交接，但不得在没有独立授权的情况下把批准请求、批准意图或批准记录用作 OpenClaw 执行依据。

## v2.7：OpenClaw 审计观察窗治理补章

### 原始阶段定位

v2.7 建立 read-only OpenClaw audit review，用于把执行面作为被观察对象纳入治理审计，而不是扩大执行面。

### 已完成内容

v2.7 已完成只读 OpenClaw 审计观察窗口，证明系统可以查看经批准的审计记录，并对执行面形成受限观察。

既有文档已经强调审计观察不得暴露 stdout、stdout_tail、approval_phrase 等敏感执行材料。

### 发现的缺陷

OpenClaw audit review 已经存在，但其治理含义需要显式框定。经批准的审计记录可能被误读为执行命令，或被误认为可以驱动 OpenClaw 自动行动。

### 本补章补齐内容

只读审计观察只能观察已批准的审计记录，不得触发 OpenClaw 执行。

经批准的审计记录不是执行命令。它们记录治理事实、风险状态、观察结果或审批链条的一部分，但不携带自动执行语义。

review tools 不得暴露 stdout、stdout_tail 或 approval_phrase。审计摘要应只呈现必要的非敏感事实和治理状态。

### 不可越界规则

观察不是执行。

### 对 v2.8.0 的约束

v2.8.0 可以把 terminal / OpenClaw validation 纳入最小闭环验证，但必须保持观察与执行分离。OpenClaw 不得仅因存在 approved audit record 而执行任何动作。

## 对 v2.8.0 的统一约束

v2.8.0 不得从扩展自动化开始。

v2.8.0 必须首先验证最小闭环：

1. natural language task -> 人类操作者给出自然语言任务、范围和边界。
2. Codex CLI implementation -> Codex CLI 在仓库内完成最小必要实现或文档变更。
3. terminal / OpenClaw validation -> 通过终端或 OpenClaw 执行受控、可复核、无越权的验证。
4. ChatGPT review -> ChatGPT 对结果、风险、边界和证据进行复核。
5. GitHub PR / tag -> GitHub 承载 PR、合并决策、tag 和版本封印。

该闭环的目标是证明治理交接，而不是增加执行权。

v2.8.0 不得暗示 OpenClaw 可以从 approved audit records 自动执行，不得暗示 Hermes memory recall 是 durable memory，不得暗示 Codex CLI 拥有 merge、tag 或 release authority。

只有在最小闭环中每个交接点都能证明证据可追溯、权限不外溢、执行不自触发、版本可回滚、人类仍为最终权威时，后续阶段才可以讨论更高层的能力扩展。
