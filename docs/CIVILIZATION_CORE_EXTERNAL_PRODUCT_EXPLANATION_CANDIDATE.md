## 3. Why External Product Explanation Matters

External readers may confuse governance documents with live product readiness.

Product language can overstate capability. A sentence that sounds useful to a product reader can accidentally imply deployment, customer use, connector readiness, API access, or production status.

Comparison language can imply dependency or adoption. External systems may provide useful methodology vocabulary without becoming dependencies, adapters, runtime behavior, or Civilization Core identity.

Connector language can imply active integration or API access. Connector governance must stay taxonomy-only unless a separate future human-approved scope changes that boundary.

Memory Graph language can imply implemented graph storage. Ontology and relationship vocabulary must stay vocabulary unless separately authorized.

`Candidate` and `sealed kernel` must remain clear. A sealed kernel is a closed governance identity, not an active commercial product. A candidate explanation is reviewable language, not public copy approval.

External explanation must separate what exists, what is documented, what is candidate, and what is prohibited.

Human review remains required before any public use, product use, connector adoption, graph behavior, P4 activation, or future implementation discussion.
