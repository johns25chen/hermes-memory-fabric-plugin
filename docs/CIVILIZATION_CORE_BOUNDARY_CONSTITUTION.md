## 4. Authority Model

The human operator retains final authority.

System output is not human approval.

Audit is not authorization.

Review is not execution.

Approval request is not approval.

Docs do not create runtime capability.

PR merge records repository change. It does not grant future authority.

A tag anchors release identity. It does not grant future mutation.

Stable kernel metadata is not active runtime.

Authority must be explicit, scoped, and human-controlled. No document, test, smoke, audit, PR, tag, candidate note, or metadata marker silently transfers authority away from the human operator.

## 5. Memory Boundary

Recall is not write.

Evidence recall is not durable memory adoption.

Candidate memory is not approved memory.

Memory proposal is not memory mutation.

Memory layer mapping is a governance coordinate. It is not live layer activation.

Layer 15 / 星源记忆 is sealed governance metadata and a methodology boundary. It is not active Star-Source Memory runtime and not active Layer 15 runtime.

Memory boundaries permit documentation, interpretation, review, and candidate comparison. They must never be interpreted to authorize durable memory writes, Memory Graph mutation, live layer activation, source mutation, or self-granted continuity.

## 6. Evidence Boundary

Evidence supports review.

Evidence is not truth by itself.

Source citation is not authorization.

Test result is not general safety proof.

Smoke result is not broad runtime permission.

Release integrity audit is not authority grant.

Evidence may establish that something was observed, cited, checked, or reproduced under stated conditions. It must never be interpreted as universal truth, broad permission, future authority, execution approval, or durable memory adoption.

## 7. Proposal / Review / Approval Boundary

Proposal is candidate only.

Review is inspection only.

Approval request is request only.

Human approval must be explicit.

Approval is scoped to the stated action.

Approval does not silently authorize unrelated future actions.

Proposal, review, and approval surfaces may prepare, inspect, reject, approve, or record a bounded decision. They must never be collapsed into one another. A proposal does not mutate. A review does not execute. A request does not approve. A scoped approval does not authorize unrelated future work.
