## 6. Candidate Object Definition and Method

A candidate object is a prose-only documentary concept used to describe a bounded unit of independently reviewable material; it is not an instantiated object, schema, record, class, entity, graph node, stored state, or runtime type.

The word object does not imply a software object, database object, stored object, graph object, executable component, or implementation commitment.

- COM-01 — A candidate object remains documentary, conceptual, non-instantiated, non-stored, non-adopted, and non-authoritative.
- COM-02 — It remains independently reviewable, human-controlled, explicitly scoped, and subject to stop conditions.
- COM-03 — Prose describes purpose and meaning without machine fields, required properties, typed attributes, runtime identifiers, or serialization.
- COM-04 — Prose creates no construction, mutation, method behavior, command, event, transition, lifecycle automation, or validation rule.
- COM-05 — Prose creates no approval, authorization, execution, persistence, or object behavior.
- COM-06 — Source grounding and provenance expose lineage without making correctness or authority claims.
- COM-07 — Assumptions, uncertainty, conflict, dissent, and gaps remain visible for qualitative human review.
- COM-08 — Each family names Han as the Human Owner review boundary and stops when scope, evidence, or separation fails.
- COM-09 — Each handoff presents documentary material for review without routing, submission automation, or follow-on work.
- COM-10 — Candidate object definitions must not become machine-readable specifications.
