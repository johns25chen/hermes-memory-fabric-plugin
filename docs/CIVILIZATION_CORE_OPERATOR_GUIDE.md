## 3. Operator Role

The operator is the human reader, maintainer, and reviewer of Civilization Core documentation.

The operator interprets evidence, boundaries, candidate status, roadmap placement, release facts, post-terminal documents, and future planning labels.

Final authority remains human.

The operator does not receive automatic authority from the system. The operator must distinguish documentation from implementation, candidate text from approved change, and release evidence from future permission.

The operator must apply these rules:

- system output is not human approval;
- docs do not create runtime capability;
- PR merge records repository change, not future authority;
- tag anchors release identity, not future mutation;
- generated documents do not approve future work;
- evidence packages do not transfer human authority.

## 5. Core Boundary Rules

Use these rules when interpreting every document, artifact, release fact, candidate note, audit, and future plan:

- recall is not write
- evidence is not truth by itself
- audit is not authorization
- review is not execution
- approval request is not approval
- adapter declaration is not dispatch
- source mutation proposal is not source mutation
- stable kernel metadata is not active runtime
- terminal marker is not roadmap continuation
- human sovereignty is non-transferable

These rules are the operator's first filter. If a later sentence appears broader than these rules, read the later sentence through the stricter rule.

## 6. Operator Decision Ladder

Before accepting any claim or continuing any post-terminal task, apply this ladder:

Step 1: Is this documentation, candidate, decision, or implementation?

Step 2: Is this inside sealed v6 or outside v6 continuation?

Step 3: Does it change version, tag, dependency, adapter, runtime, writer, Memory Graph, authorization, or execution?

Step 4: Does it require explicit human approval?

Step 5: If uncertain, stop and classify as candidate only.

Uncertainty must fail closed.

Ambiguous claims must be downgraded to candidate / documentation-only status.

There is no hidden escalation from candidate to capability.

## 7. Evidence Handling

Evidence supports review.

Evidence must be cited or traceable when used.

Evidence can be stale, incomplete, scoped, or tied to a specific command, file, branch, tag, PR, test run, smoke run, audit run, or document revision.

Tests, smokes, and audits are bounded evidence. A passing result shows that the checked condition passed under the checked scope. It does not prove broad future safety.

Evidence does not authorize execution.

Evidence does not create durable memory.

Release evidence records what happened or what was verified. It does not create future authority, reopen a sealed line, activate a runtime, or approve unrelated work.

## 8. Recall and Memory Handling

Recall is retrieval or resurfacing.

Recall does not write.

Candidate memory is not approved memory.

Compiled knowledge is not approved memory.

Associative recall candidate is not durable memory adoption.

Review queue is not approval.

Layer 15 / 星源记忆 remains governance metadata and methodology boundary, not active runtime.

When recalled content is useful, the operator may use it as historical context. The operator must still verify scope, freshness, and authority before treating it as current evidence.

Memory layer descriptions are governance coordinates. They do not activate live layers, durable writers, Memory Graph mutation, source mutation, self-evolution authority, approval engines, or executors.

## 9. Proposal / Review / Approval Handling

Proposal = candidate.

Review = inspection.

Approval request = request.

Approval = explicit scoped human decision.

Approval does not silently authorize unrelated future work.

Approval cannot be inferred from PR merge, tag, test pass, or generated document.

The operator must preserve the separation:

| Surface | Operator interpretation |
| --- | --- |
| Proposal | A candidate has been formulated. |
| Review | The candidate has been inspected. |
| Approval request | Human authority has been requested. |
| Approval | A human made an explicit scoped decision. |
| Merge | Repository content changed. |
| Tag | Release identity was anchored. |
| Test pass | A bounded check passed. |
| Generated document | Text was produced. |

None of these surfaces silently becomes another surface.

## 10. Execution Handling

There is no automatic execution.

There is no hidden external action.

There is no network action without explicit operator control.

There is no durable write without governed approval.

There is no Memory Graph mutation without governance.

Operation contract is not dispatch.

Doctor report is not remediation.

Eval pass is not authorization.

Execution-related language must be interpreted as boundary language unless a separate, explicit, human-approved implementation scope exists outside the sealed v6 line. In this T6 context, no such implementation scope exists.

External projects are methodology candidates only.

## 14. Stop Conditions

The operator must stop immediately if any of these appear:

- unexpected source/test/script/pyproject change
- version changed from 6.16.0
- uv.lock appears
- v6.17 appears
- dependency install instruction appears
- runtime connection wording appears
- adapter implementation appears
- Memory Graph mutation appears
- durable writer appears
- authorization/execution semantics appear
- external project becomes identity or completion condition
- active Layer 15 runtime claim appears
- active Star-Source Memory runtime claim appears
- autonomous authority / self-authorization / personhood / life / awakening / legal subject / religious status claim appears

Stopping means no continuation, no cleanup-by-implementation, and no silent reinterpretation. Classify the issue, preserve evidence, and return to human review.

## 15. Operator Checklist

Before continuing any post-terminal task, confirm:

- on correct branch
- main is clean before branch creation
- version remains 6.16.0
- no uv.lock
- docs-only scope
- one intended file only
- no tag for docs-only work
- no v6.17
- external candidates remain candidates
- no runtime / dependency / adapter / writer / Memory Graph mutation / authorization / execution semantics

If any checklist item fails, stop and classify the work as blocked until the human operator reviews it.
