## 8. Candidate-by-Candidate Review

### llm_wiki

Safe interpretation: `llm_wiki` is a knowledge compiler methodology candidate.

Useful methodology: raw source preservation, compiled knowledge candidates, purpose/schema/index/log discipline, review queues, lint health, graph insight, source-aware packaging, and documentation-to-context compression.

Main drift risk: readers may treat compiled knowledge or graph insight as approved memory, dependency adoption, automatic ingest, or Memory Graph mutation.

Current boundary protection: existing docs classify `llm_wiki` as methodology only, not implemented, not a dependency, not an adapter, not a v6 completion condition, and not Civilization Core identity.

Future review question: can future design-only discussion borrow source-aware knowledge packaging without creating ingest, dependency, adapter, writer, graph mutation, or approval behavior?

Explicit non-authorization statement: this comparison does not authorize importing `llm_wiki`, adding it as a dependency, creating an adapter, activating compiled wiki behavior, mutating Memory Graph, writing durable memory, starting v7 implementation, or building product behavior.

### M-Flow

Safe interpretation: M-Flow is an associative recall methodology candidate.

Useful methodology: Episode, Facet, FacetPoint, Entity, semantic edge, path-cost recall, bundle search, associative evidence paths, multi-hop memory association, and relevance path explanation.

Main drift risk: readers may treat recall graph thinking as active retrieval, durable memory adoption, graph mutation, dependency adoption, or adapter work.

Current boundary protection: existing docs classify M-Flow as candidate-only material outside `v6.16.0`, not runtime behavior, not a dependency, not an adapter, not Memory Graph mutation, not authorization, not execution, not active Star-Source Memory, not active Layer 15, and not a v6 completion condition.

Future review question: can future design-only discussion borrow associative recall vocabulary and relevance-path explanation without creating vector retrieval, graph retrieval, adapter behavior, durable writes, or authorization semantics?

Explicit non-authorization statement: this comparison does not authorize importing M-Flow, adding a dependency, creating an adapter, activating recall behavior, mutating Memory Graph, writing durable memory, starting v7 implementation, or building product behavior.

### GBrain

Safe interpretation: GBrain is a memory operations methodology candidate.

Useful methodology: answer + citations + gaps, Brain / Source boundary, operation contract vocabulary, localOnly / remote / scope trust boundary, doctor / health report, eval / replay / benchmark, lifecycle language, and operator-facing workflow concepts.

Main drift risk: readers may treat operations vocabulary as MCP/API activation, dispatch, admin behavior, write behavior, remote operation, automatic remediation, MVP, or Operator Console implementation.

Current boundary protection: existing docs classify GBrain as methodology only, not implemented, not a dependency, not an adapter, not an MCP/API operation surface, not dispatch, not remediation, not authorization, and not execution.

Future review question: can future design-only discussion borrow operations vocabulary and trust-boundary thinking without creating an operation surface, write/admin behavior, remote action, authorization, execution, or product deployment?

Explicit non-authorization statement: this comparison does not authorize importing GBrain, adding a dependency, creating an adapter, activating operation behavior, creating MCP/API surface, mutating Memory Graph, writing durable memory, starting v7 implementation, or building product behavior.
