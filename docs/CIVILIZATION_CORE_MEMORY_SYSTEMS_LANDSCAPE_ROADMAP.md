## 3. Why This Roadmap Exists

The human operator wants broad future research into open-source AI memory, brain, second-brain, and agent-memory systems across GitHub and related public sources.

This roadmap exists before that research begins so the future report cannot drift into implementation, productization, dependency adoption, adapter work, runtime behavior, v7 work, or identity transfer.

The future report may study external projects, but it must not absorb external runtime, code, dependencies, adapters, product surfaces, operation surfaces, or identity into Civilization Core.

The research output must feed methodology candidates and Civilization Core shortboard mapping only.

P4 remains paused unless separately confirmed by the human operator.

## 4. Research Boundary

| Area | Included? | Reason | Boundary |
| --- | --- | --- | --- |
| Agent memory | yes | Directly relevant to governed recall, write, evidence, and approval separation. | Study methodology only; do not adopt agent runtime, tools, or write behavior. |
| LLM memory | yes | Relevant to long-term context, retrieval, update, and safety models. | Treat memory claims as candidates, not approved architecture. |
| AI second brain | yes | Relevant to personal and team knowledge continuity patterns. | Do not convert product metaphors into Civilization Core identity or MVP scope. |
| Personal AI memory | yes | Relevant to human-controlled memory governance and privacy posture. | No durable writer, product deployment, or personal data workflow follows. |
| Memory operating system | yes | Useful for lifecycle, governance, and operator vocabulary. | Operation vocabulary must not become an operation surface. |
| Knowledge graph memory | yes | Relevant to provenance, explainable recall, and relationship modeling. | Graph ideas must not become Memory Graph mutation. |
| RAG memory | limited | Useful when it has explicit memory lifecycle, provenance, or governance. | Generic retrieval stacks are excluded unless memory-specific. |
| Long-term memory agent | yes | Relevant to recall/write distinction and persistence risks. | No autonomous authority, self-authorization, or execution semantics. |
| AI brain | limited | Useful when architecture is explicit and memory-focused. | Brain language must not imply life, personhood, awakening, legal status, or religious status. |
| Enterprise AI search | limited | May provide governance, provenance, and evaluation patterns. | Search-only products are excluded unless memory governance is central. |
| Generic chatbot UI | no | UI chat alone does not establish memory-system methodology. | Exclude unless memory model, governance, and lifecycle are primary. |
| Pure vector database | no | Storage infrastructure alone is not a memory governance system. | May be mentioned only as dependency risk, not as candidate essence. |
| Closed-source commercial product | no | The future report targets open-source projects with inspectable artifacts. | May be contextual background only if explicitly scoped later. |
| Paper-only project | limited | Papers may explain methodology but do not provide repo health signals. | Do not include in top 10 repo rankings unless implementation exists. |
| Benchmark-only repository | limited | Benchmarks may inform evaluation. | Do not rank as memory system unless it includes a memory architecture. |
| Awesome list / survey repository | limited | Useful for discovery leads. | Do not rank as a project unless it implements a system. |

## 7. Popular Top 10 Selection Method

The future report may select a popular top 10 only after live research.

Selection criteria must include:

- high GitHub stars;
- meaningful forks;
- mature README;
- active maintenance;
- clear memory / brain relevance;
- ecosystem impact;
- real usage signals;
- not just generic RAG or chatbot;
- not selected only for hype.

The future report must cite current data at research time. Stars, forks, last commit date, release activity, and issue / PR activity must not be copied from this roadmap because this roadmap performs no search and produces no ranking.

Popularity is evidence of attention, not governance approval.

The future report may select a rising top 10 only after live research.

- recent creation or recent growth;
- recent commits;
- active issues / PRs;
- emerging architecture pattern;

## 10. Essence Extraction Framework

| Essence module | What it means | Why it matters | Likely source projects | How it can help Civilization Core | What must not be inferred |
| --- | --- | --- | --- | --- | --- |
| Memory Evaluation | Methods for judging recall quality, write quality, staleness, and failure modes. | Civilization Core needs stronger evidence for memory quality. | Benchmarks, eval suites, memory agents with replay. | Feed future evaluation-candidate docs and P4 checklist material. | Benchmark pass is not governance approval. |
| Temporal Validity | How memories age, expire, conflict, or remain valid across time. | Prevents stale memory from becoming false authority. | Temporal graph, evolving knowledge, lifecycle repos. | Inform temporal validity model candidates. | Time metadata is not automatic correction or write permission. |
| Provenance | Source, citation, log, hash, and trace evidence for memory claims. | Keeps recall explainable and reviewable. | Knowledge graph, source-aware memory, local-first systems. | Strengthen source-of-truth and citation discipline. | Provenance is not truth by itself. |
| Confidence | Confidence, uncertainty, gap, or quality scoring around memory. | Helps humans review candidate memory without overtrust. | Eval-heavy and explainable recall projects. | Inform confidence taxonomy and shortboard mapping. | Confidence score is not approval. |
| Lifecycle | Candidate, reviewed, approved, rejected, stale, deleted, and archived states. | Separates memory states and prevents silent promotion. | Memory operating systems and operator tools. | Feed lifecycle taxonomy candidates. | Lifecycle labels are not durable writes. |
| Failed Attempt Memory | Recording failed attempts, blocked paths, and do-not-repeat evidence. | Prevents repeated unsafe or useless attempts. | Agent memory, replay, benchmark, and ops projects. | Inform failed-attempt memory / do-not-retry rules. | Failure records must not become automatic prohibition without review. |
| Explainable Recall | Showing why a memory was retrieved and which evidence path supported it. | Makes recall inspectable before action. | Graph memory, associative recall, citation systems. | Strengthen recall trace templates. | Explanation is not authorization. |
| Connector Governance | Controls around integrations, tools, APIs, and data sources. | Prevents connector enthusiasm from becoming unsafe operation. | Memory OS, enterprise tools, agent frameworks. | Inform connector governance taxonomy. | Connector list is not adapter approval. |
| Human Review Gate | Explicit human inspection before adoption, mutation, or action. | Preserves human authority. | Governance-first memory tools and review-queue systems. | Feed P4 decision gate checklist. | Review gate is not approval by itself. |
| Markdown Source of Truth | Human-readable docs as source, schema, log, and review surface. | Keeps methodology auditable and portable. | Knowledge compiler and doc-first systems. | Improve future docs without changing runtime behavior. | Markdown is not durable memory writer. |
| Ontology / Taxonomy | Named categories for memory kinds, states, roles, and relationships. | Reduces ambiguity in future design discussion. | Knowledge graph, memory OS, AI brain systems. | Inform ontology / taxonomy weakness mapping. | Taxonomy is not live layer activation. |
| Product Mental Model | How a project explains user value and user workflows. | Helps Civilization Core become understandable without productization drift. | Second-brain, personal AI memory, enterprise knowledge tools. | Improve external explanation and narrative candidates. | Product metaphor is not MVP or deployment approval. |
